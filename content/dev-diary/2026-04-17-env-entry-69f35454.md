### この回の要点（先に読む）

- **変わったこと**: 「ツールを最新にする」と「AIモデルが変わったことで運用自体を見直す」を、別の管理サイクルとして明確に分けた
- **いままでとの違い**: バージョンが上がったら「何となく対応する」という曖昧な運用だった
- **メリット**: 「ツールは最新なのに、使い方が古いまま」という状態が起きなくなった

### 何が変わったか

#### 炊飯器で例えると分かりやすい

```
ツールのアップデート
  = 炊飯器が少し早く炊けるようになった
  → 同じ手順で少し速い。段取りは変わらない

開発環境の運用見直し
  = 予約タイマーが付いた
  → 「帰宅後に研いでから炊く」という前提がなくなった
  → 「前の晩にセットして朝炊き上がり」に段取りが変わった
  → 道具の性能が上がったことで、生活リズム自体が変わった
```

開発環境に置き換えると：

```
ツールのアップデート（週次・月次）
  = Claude Code が v2.1 → v2.2 になった
  → 同じ使い方で少し速く・正確になった。進め方は変わらない

開発環境の運用見直し（60日周期 + イベント駆動）
  = Opus 4.7 が登場し、判断精度が大きく上がった
  → 「Opus はコストが高いから設計でも Sonnet で済ませる」
    という前提そのものを見直せるようになった
  → モデルの性能が上がったことで、運用ルール自体が変わった
```

「ツールを最新にする」は **道具を整える** こと。
「運用を見直す」は **道具が変わったことで、やり方・考え方を変える** こと。
今まで前者はやっていましたが、後者は「バージョンが上がったら何となく対応する」という曖昧な運用でした。

---

#### 3層の管理サイクルとして整理した

今回、これを明示的に3層のサイクルとして定義しました。

```
週次 /changelog
  「Claude Code 自体に何か新機能・変更はあったか？」
  → 道具の変化を追う

月次 /tool-audit
  「使っているツール群（Gemini CLI, Codex CLI 等）に更新はあったか？」
  → 道具全体を点検する

60日周期（+ イベント駆動） /methods-audit
  「AIモデルが変わったことで、やり方そのものを見直すべきか？」
  → 運用・考え方を再検証する
```

特に `/methods-audit` は、主要モデルのバージョンアップ（今回の Opus 4.7 登場など）があれば60日を待たず前倒しで実施するルールにしました。

### 今回対応したこと

- ✅ `/methods-audit` スキル新設（60日周期 + イベント駆動）
- ✅ 週次・月次・60日の3層管理サイクルを確立・文書化
