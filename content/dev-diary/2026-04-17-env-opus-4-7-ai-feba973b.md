### この回の要点（先に読む）

- **変わったこと**: 「作業の重さに応じてモデルを変える」という感覚的な運用を、「正しい意思決定 1回あたりのコスト」という明確な基準でルール化した
- **いままでとの違い**: どの場面でどのモデルを使うかが感覚任せだった
- **メリット**: 判断の場面でモデル選択に迷わなくなった。重要な設計でケチって手戻りが起きるリスクが減った

### 何が変わったか

#### Opus 4.7 の登場がトリガーになった

Claude の最上位モデルが 4.6 から **4.7** に更新されました。

このとき「モデルの番号を書き換えるだけ」ではなく、それを契機にモデルの使い方そのものを見直しました。

もともと「作業の重さに応じてモデルを使い分ける」という考えはありました。ただ「どの場面が重いのか」「どこまでケチっていいのか」の基準は感覚的でした。

今回、その基準を言語化しました。

```
正しいコスト管理は「トークン単価」ではなく
「正しい意思決定 1回あたりのコスト」で考える

  たとえば：
  設計の判断を、コスト節約のため下位モデルで済ませる
    → 判断にズレが生じる
    → 実装が完成した後に発覚
    → やり直し発生

  「安く抑えた」はずが、手戻りで結果的に高くついた
  重い判断ほど、上位モデルを使う方がトータルで安い
```

---

#### モデル使い分けのルールを明文化した

```
┌────────────────────────────────────────────────────┐
│         モデル使い分けの基準                          │
├────────────────────────────────────────────────────┤
│                                                    │
│  重い意思決定（設計・アーキテクチャ・トレードオフ比較）   │
│    → Opus 4.7（Plan mode）                         │
│    ※ 正確な判断が手戻りを防ぐ。ここはケチらない        │
│                                                    │
│  通常の実装・小修正・テスト修正                        │
│    → Sonnet 4.6（デフォルト）                       │
│                                                    │
│  広い検索・コードベース探索                            │
│    → Haiku 4.5（サブエージェント）                   │
│                                                    │
└────────────────────────────────────────────────────┘
```

また「こういう状況になったら Opus に切り替える」というシグナルも整備しました。

```
自動エスカレーションのシグナル
  🟡 同じ論点で2回以上アプローチを変えている（堂々巡り）
  🟡 選択肢 A / B のどちらを推すか明確にできない（比較不能）
  🟡 実装後に設計の前提が崩れた（前提崩壊）
  🔴 医療・認証・DB変更など高ステークスの設計判断
```

### 今回対応したこと

- ✅ コスト管理フレームを「トークン単価」→「正しい意思決定あたりコスト」に全面刷新
- ✅ `opusplan` 設定導入（Plan mode のみ Opus 4.7、実行フェーズは Sonnet 4.6）
- ✅ 自動エスカレーション検知シグナル表を整備（堂々巡り・比較不能・前提崩壊・高ステークス）
- ✅ methods/ 全体の Opus 4.7 対応（コスト管理レポート・運用ガイド・テンプレート更新）
