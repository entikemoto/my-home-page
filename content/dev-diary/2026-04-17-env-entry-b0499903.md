### この回の要点（先に読む）

- **変わったこと**: AIモデルの使い方を「なるべく安いモデルで」から「正しい意思決定に最適なモデルで」に切り替えた
- **いままでとの違い**: ツールのバージョンアップと、開発環境全体の見直しを「別のこと」として管理するようになった
- **メリット**: 安いモデルを使って判断を間違えるより、適切なモデルで正しく判断する方が総コストが下がる、という考え方が仕組みに入った

### 何が変わったか

#### コスト管理の「考え方」を変えた

今まで「コストを下げる＝なるべく安いモデルを使う」と思っていました。

でもこれは間違いでした。

たとえばこういうことが起きます。

```
安いモデルで設計を決める
  → 設計に欠陥があった
  → 実装が完成した後に発覚
  → 全部やり直し

「安く」しようとした結果、手戻りで3倍のコストがかかった
```

正しいコスト管理は「トークン単価」ではなく「正しい意思決定 1回あたりのコスト」です。

```
┌────────────────────────────────────────────────────┐
│         コスト管理の考え方の変化                       │
├────────────────────────────────────────────────────┤
│                                                    │
│  旧：なるべく安いモデルを使う                          │
│      ↓ 安いが判断を間違える                           │
│      ↓ やり直しが発生                                │
│      ↓ 結果的に高くつく                              │
│                                                    │
│  新：正しく判断できるモデルを使う                       │
│      ↓ 高く見えるが判断が正確                         │
│      ↓ やり直しが起きない                             │
│      ↓ 結果的に安くつく                              │
│                                                    │
└────────────────────────────────────────────────────┘
```

具体的には、**設計・アーキテクチャ判断・トレードオフ比較**などの「重い意思決定」には Claude の最上位モデル（Opus 4.7）を使い、**単純な実装・小修正・ファイル検索**などの「軽い作業」は中位・下位モデルで行うルールを整備しました。

---

#### 「ツールのアップデート」と「開発環境全体の見直し」は別物だった

これは今まであまり意識していなかった区別です。

「開発ツールのアップデート」と「開発環境全体の見直し」は、役割が根本的に違います。

```
ツールのアップデート（定期的に実施）
  ↓
  「使っているツールが古くなっていないか？」
  例: Claude Code が v2.1 → v2.2 になった
  例: Gemini CLI に新コマンドが追加された
  → 個別のツールをひとつずつ最新化する

開発環境全体の見直し（60日に1回 + イベント駆動）
  ↓
  「AIモデルが変わったことで、今の進め方そのものが最適か？」
  例: Opus 4.7 が登場 → 今まで「Opus は高いから避ける」が正しかったか？
  例: 新モデルの能力が上がった → プロンプトの書き方を変えるべきか？
  → やり方・考え方・ルールそのものを再検証する
```

簡単に言うと：

- ツールのアップデート ＝ **道具を最新にする**（週次・月次）
- 開発環境全体の見直し ＝ **やり方・考え方を最新にする**（60日周期）

今まで前者はやっていましたが、後者は「Claudeのバージョンが上がったら何となく対応する」という曖昧な運用でした。今回、これを明示的に `/methods-audit` というスキルとして定義し、**60日に1回、または主要モデルのバージョンアップ（今回の Opus 4.7 登場のような出来事）があれば前倒しで実施する**ルールにしました。

---

#### 今回のトリガー：Opus 4.7 の登場

Claude の最上位モデルが 4.6 から **4.7** に更新されました。

このとき「モデルの番号を書き換えるだけ」ではなく、それを契機に開発環境全体の見直しを行いました。

具体的に見直したのは：

- 「Opus はコストが高いから使わない」という前提がまだ valid か？→ **前提ごと変えた**（上述のコスト管理見直し）
- Plan mode（設計フェーズ）でだけ Opus を使い、実装フェーズは Sonnet を使う `opusplan` という設定を導入
- 「AIがどんな状況でモデルを切り替えるべきか」のシグナル一覧を整備

```
自動エスカレーションのシグナル（こうなったら Opus に切り替え）
  🟡 同じ論点で2回以上アプローチを変えている（堂々巡り）
  🟡 選択肢 A / B のどちらを推すか明確にできない（比較不能）
  🟡 実装後に設計の前提が崩れた（前提崩壊）
  🔴 医療・認証・DB変更など高ステークスの設計判断
```

---

#### まとめ：3層の管理サイクルができた

```
週次 /changelog
  「Claude Code 自体に何か新機能・変更はあったか？」

月次 /tool-audit
  「使っているツール群（Gemini CLI, Codex CLI 等）に更新はあったか？」

60日周期（+イベント駆動） /methods-audit
  「AIモデルが変わったことで、やり方そのものを見直すべきか？」
```

この3層があることで、「ツールは最新なのに、使い方が古いまま」という状態が防げます。

### 今回対応したこと

- ✅ コスト管理フレームを「トークン単価」→「正しい意思決定あたりコスト」に全面刷新
- ✅ `opusplan` 設定導入（Plan mode のみ Opus 4.7、実行フェーズは Sonnet 4.6）
- ✅ 自動エスカレーション検知シグナル表を整備（堂々巡り・比較不能・前提崩壊・高ステークス）
- ✅ `/methods-audit` スキル新設（60日周期 + イベント駆動）
- ✅ 週次・月次・60日の3層管理サイクルを確立
- ✅ methods/ 全体の Opus 4.7 対応（コスト管理レポート・運用ガイド・テンプレート更新）
