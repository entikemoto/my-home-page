### この回の要点（先に読む）

- **変わったこと**: Claude Code（AIアシスタント）が「Docker Sandboxes」という隔離された環境の中で動くようになった。
- **いままでとの違い**: これまでClaude Codeは自分のMac本体の上で直接動いていた。万が一AIが誤ったコマンドを実行しても、Mac全体に影響が出る可能性があった。
- **メリット**:
  - パスワードやSSHキーなどの機密ファイルに、AIが一切触れなくなった
  - 万が一AIが危険なコードを実行しても、「箱」の外には影響が出ない
  - 次回から `sbx run claude` の1行で、安全な環境が起動する

---

### そもそも「Docker（ドッカー）」とは何か

#### コンテナ船のたとえ

「Docker」という名前は、港で働く**波止場の労働者（Docker）**と、**コンテナ（荷物の箱）**から来ています。

現代の国際貿易では、どんな荷物でも同じ規格の「コンテナ」に入れて船で運びます。このコンテナの発明によって、「中身が何であっても同じクレーンで積み降ろしできる」「どの船にも積める」という革命が起きました。

ソフトウェアの世界でも同じことが起きました。**Dockerは「ソフトウェアのコンテナ化技術」**です。

プログラムを動かすには、そのプログラムが必要とする「環境」（使用するプログラミング言語のバージョン、ライブラリ、設定など）がすべて揃っていないと動きません。「自分のパソコンでは動いたのに、別のパソコンでは動かない」という問題が昔からありました。

Dockerは、**プログラムと、そのプログラムが必要とするすべての環境を、ひとつの「箱（コンテナ）」にまとめます**。この箱ごと渡せば、どのパソコンでも同じように動きます。

#### アパートのたとえ

もう少しわかりやすく言うと、Dockerは**「一棟のビルの中に、独立したアパートをいくつも作る技術」**です。

あなたのMacは「土地」です。その上に「ビル（OS）」が建っています。Dockerは、そのビルの中に「完全に独立したアパート（コンテナ）」をいくつも作れます。

各アパートは：
- 他のアパートの住人が何をしているか見えない
- 共有の廊下（CPU・メモリ）は使うが、部屋の中は完全に独立
- アパートを解体しても、ビル（Mac本体）には何も残らない

### 今回使った「Docker Sandboxes（サンドボックス）」とは

Dockerの「コンテナ」よりさらに強力な隔離技術が「**サンドボックス**」です。

「サンドボックス」とは、直訳すると**砂場**です。子どもが砂場で何をしても、庭の芝生は汚れません。砂場の中だけの世界です。

コンピュータの世界でも同じ考え方です。**「サンドボックスの中で何が起きても、外には影響しない」**という安全地帯のことをサンドボックスと呼びます。

今回設定した「Docker Sandboxes」は、コンテナよりさらに強力な**マイクロVM（超小型の仮想コンピュータ）**を使っています。

- **通常の Claude Code** — 隔離なし。家の中で直接作業するイメージ
- **Docker コンテナ** — 中程度の隔離。部屋を仕切る壁
- **Docker Sandboxes（今回）** — 最強の隔離。別の建物に移動するイメージ

**別の建物**に移動すると、たとえ建物が火事になっても、自宅（Mac本体）には燃え移りません。

### 何が守られるのか

#### 守られるもの①：機密ファイル

Macの中には、様々な「鍵」が保存されています。

- `~/.ssh/` ── GitHubやサーバーへのログインに使う鍵
- `~/.aws/` ── AWSクラウドへのアクセスキー
- `~/.zshrc` ── ターミナルの設定（APIキーが書かれていることも）

サンドボックスの中のClaude Codeは、**これらのファイルが存在することすら知りません**。「箱の外」にあるものは完全に見えない状態です。

#### 守られるもの②：ネットワーク

通常のClaude Codeはインターネットに自由にアクセスできます。悪意あるコードが「あなたのコードを外部サーバーに送信する」ことも、理論上は可能でした。

Docker Sandboxesでは、**接続を許可したドメイン（例: github.com）以外への通信は、すべてブロックされます**。

#### 守られるもの③：Mac本体の安定性

もしClaude Codeが無限ループするプログラムを書いたとしても、サンドボックスにCPU・メモリの上限が設定されているため、**Mac本体がフリーズすることはありません**。

### セットアップの全手順（再現用）

```bash
# ステップ1: sbx CLIをインストール（初回1回だけ）
brew install docker/tap/sbx

# ステップ2: Dockerアカウントにログイン（初回1回だけ）
sbx login

# ステップ3: Anthropic APIキーを登録（初回1回だけ）
sbx secret set ANTHROPIC_API_KEY
# → 実行するとパスワード入力欄が出る。そこにAPIキーを貼る（画面には表示されない）

# ステップ4: 動作確認
sbx run claude
```

### 日常の運用方法

#### 毎回の起動手順（ステップ4以降）

新しいプロジェクトで作業するときは、そのフォルダに移動してから起動するだけです。

```bash
# 1. 作業したいプロジェクトフォルダに移動
cd "/Users/takeshiikemoto/Obsidian Vaults/MainVault/projects/20260216_VoiceClinical"

# 2. サンドボックスを起動
sbx run claude

# → 自動的にそのフォルダが「箱の中」にマウントされ、Claude Codeが起動する
```

#### よく使うコマンド一覧

```bash
# 起動中のサンドボックスを確認する
sbx ls

# サンドボックスを止める（データは残る）
sbx stop claude-<プロジェクト名>

# サンドボックスを完全に削除する
sbx rm claude-<プロジェクト名>

# 登録済みのSecretを確認する
sbx secret list

# Secretを更新する（APIキーを変えたとき）
sbx secret set ANTHROPIC_API_KEY
```

#### 現在の状態を確認する

```bash
sbx ls
```

実行すると以下のような表が表示されます：

```
SANDBOX            AGENT    STATUS    WORKSPACE
claude-MainVault   claude   running   /Users/.../MainVault
```

- `running` ── 現在動いている
- `stopped` ── 停止中（`sbx run claude` で再起動できる）

#### Q. 毎回 `sbx run claude` を使わないといけないの？

**A. 普通の作業には必須ではありません。**

`claude` コマンドで起動する通常のClaude Codeも引き続き使えます。「信頼できないコードを実行する」「外部から持ってきたコードを試す」「セキュリティが特に重要な作業をする」ときだけ `sbx run claude` を使うのが現実的な運用です。

- 普段の開発・執筆・調査 → `claude`
- 外部コードの実行・テスト → `sbx run claude`
- 医療データを扱う作業 → `sbx run claude`
- 新しいライブラリを試す → `sbx run claude`

### 中嶋聡さんの体験と同じことが起きた

中嶋聡さんは「Dockerを使ったコードを書いたことがなかったので、Claude Codeから教わりながら指示を出すという不思議な体験だった」と話しています。

今回も同じでした。`sbx` のインストールから認証、動作確認まで、Claude Codeと対話しながら進めました。最終的に確認できたのが：

```
SANDBOX            AGENT    STATUS
claude-MainVault   claude   running
```

この `running` の表示が、すべてが正しく動いていることの証明です。

**AIに教わりながら、AIを安全に動かす環境を作る。**

数年前なら専門のエンジニアが数日かけてやっていたセキュリティ設定が、対話形式で数十分で完了する時代になりました。
