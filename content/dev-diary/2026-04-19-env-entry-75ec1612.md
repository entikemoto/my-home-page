### この回の要点（先に読む）

- **変わったこと**: 「AIエージェントが勝手に変なことをしていないか」を自動で見張る仕組みが3層入った
  - 層1: 外部（WebやGmailやGitHub）から読んだデータを使って直後にネットに送ろうとしたら警告が出る
  - 層2: 作業ごとに「今回は curl 禁止・Gmail 禁止」みたいなチケットを書いておくと、違反したときにブロックされる
  - 層3: 「本当に守れているか」を攻撃者の気持ちでテストする自動テストが週1回動く
- **いままでとの違い**: これまでは「allow/deny を settings.json に書く」の1層だけだった。作業ごとに絞る手段がなく、外から読んだ指示に釣られて変な送信をしても気づけなかった
- **メリット**: これまでの弱点だった「hook だけに頼る守り」を見直せた。hook とは、AI がツールを使う前後に動く見張り番のような仕組みで、「その操作は危ない」と警告したり止めたりできるが、それだけでは本当の防火壁にはならない。AI はローカル PC 上で動くため、保存済みの設定、認証情報、ローカルファイルなどに思った以上に近い位置にいる。この前提で、作業ごとの権限絞り込み・出口の監視・自動テストまで含めた多層防御に広げた。さらに、外部ツールや依存関係の新しい脆弱性を毎日自動で点検するようにした

### 具体的に何ができるようになったか

- **日次** 7:30 に OSV.dev（脆弱性データベース）で MCP サーバーや依存パッケージの新規 high/critical アドバイザリー（危険度が高い注意情報）を比較する。検出があれば翌セッション開始時に最優先で知らせてくれる
- **週次** 月曜 9:15 に「不変条件（Invariant = 絶対に守られているべき条件）」を11項目チェックする。`settings.json` の権限、secret パターン、hook の登録状態、logs ディレクトリの権限、各プロジェクトの deny など。違反があればセッション開始時に注入する
- **週次** 月曜 9:30 に赤チーム（攻撃者視点）テスト19本を走らせて、hook が期待通りブロック/警告するか検証する。19/19 合格を継続できるかを監視する
- **作業ごと**に `.claude/ticket.yaml` という作業許可証を書けば、その作業では「curl 禁止」「Gmail 禁止」「PR 作成禁止」などを宣言的に指定できる。期限切れ（例: 2時間）になると自動で全ブロックに切り替わる
- **外部データ取込後の連鎖**（WebFetch 直後に curl でどこかに送る等）を自動で検知して警告を注入

### なぜ必要だったか

- Codex GPT-5.4 に設計レビューをお願いしたところ、「危ない動きを見つけて警告する仕組み」は増えていたが、「最初からその危ない操作をできなくしておく仕組み」がまだ弱い、と分かった
- また、「情報が最終的にどこへ出ていくのか」を一覧で管理できていなかった。読む・整形する・保存する・送る、の流れを見える化しないと、思わぬ出口を見落としやすい
- 医療AIの領域では患者情報を扱うため、「Claude がうまく気をつけてくれるはず」に頼るのでは足りない。外へ送れる道そのものを、先に閉じておく必要がある
- さらに、hook を増やすだけだと「設定したつもりだが実際には動いていない」という状態が起きやすい。そのため、攻撃者の立場で試す自動テストを毎週回し、守りが本当に効いているかを確認する必要があった

### 動作確認

- capability-ticket-guard（作業許可証を見てブロックする見張り）: 10ケース全合格（Bash curl/WebFetch/mcp github write → block、mcp github read → pass、期限切れチケット → 全 block、チケットなし → pass-through〔素通し〕）
- sink-provenance-guard（外部データの直後の送信を警戒する見張り）: 9ケース全合格（外部取込後の curl/scp/rsync/eval/MCP write → warn、vault 内書込 → pass、取込なし → pass、120s 超過 → pass）
- Invariant Checker 初回実行: `~/.claude/settings.json` のパーミッション 644 → 600 修正、既存13プロジェクトの `.claude/settings.local.json` 欠落を検出（今後順次対応）
- 設計書: `docs/security-review/workspace-separation-design.md`（PHI を vault 外へ）と `docs/security-review/os-task-launcher-design.md`（data-class 別プロファイル）はユーザー確認待ち
