## 1. 業務プロセス（現状）

```
① CortexFlow2.0 で朝3本・夜3本の解説記事を作る
       ↓
② note では、3本を1つの記事にまとめて配信する
       ↓
③ X / Slack でも現在の方式で配信する
       ↓
④ 後日「あの記事どこだっけ」と思っても、個別記事として探しにくい
       ↓
⑤ 個人のプロフィール・思想・活動も、同じ場所では十分に見せられていない
```

---

## 2. 現状の課題

### 読者側の課題

| 問題 | 詳細 |
|------|------|
| **個別記事を探しにくい** | note の1記事に3本が入っているため、1本ずつ再訪しづらい |
| **検索性が弱い** | タイトル・日付・タグで絞り込んで探す導線が弱い |
| **過去記事の蓄積が見えにくい** | 日々の発信は続いているが、資産として一覧しづらい |

### 発信者側の課題

| 問題 | 詳細 |
|------|------|
| **個人ブランドの受け皿が弱い** | note / X / Slack は配信には強いが、プロフィールや思想、今後の医療AI活動をまとめて見せにくい |
| **媒体ごとの役割が曖昧** | どこが正本で、どこが配信面なのかが整理されていない |
| **HP追加時に運用が増えやすい** | 別原稿が必要になると、日々の6本運用に負荷がかかる |

## 3. このサービスが変える部分

```
① CortexFlow2.0 で朝3本・夜3本の解説記事を作る   ← 従来通り
       ↓
② note では3本まとめ記事として配信する         ← 従来通り
       ↓
┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
┃ ③ HPでは、同じ3本を1本ずつ個別記事として保存・表示する ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
       ↓
④ 読者はタイトル・日付・タグから個別記事を探せる
       ↓
⑤ 同じHP内でプロフィール・Vision・将来の医療AI活動も見せられる
```

### Before / After まとめ

| 観点 | Before（現状） | After（このサービス導入後） |
|------|----------------|------------------------------|
| **記事の保存単位** | note の3本まとめが中心 | HPでは1本ずつの個別記事 |
| **検索性** | 個別記事を探しにくい | タイトル・日付・タグで探せる |
| **媒体の役割** | note / X / Slack 中心で分散 | HPが個別記事の正本、note はまとめ、X / Slack は配信導線 |
| **ブランド表現** | 発信はあるが集約点が弱い | HPを個人ブランドの母艦にできる |

### 一言でいうと

> **「note の中の3本を、HPでは1本ずつ探しやすく見せる個人ブランドサイト」**

---

## 3.5 運用方針の明確化

- **HP内で個別記事ページを持つ**
  - HPから note のまとめ記事へ飛ばすだけにはしない
  - 1本ずつの解説記事は、HP内で完結して読めるようにする
- **note は今まで通り3本まとめ記事**
  - 朝便・夜便としての読み味は維持する
  - 必要なら HP 個別記事へのリンクを補助的に置く
- **URLは役割を分けて持つ**
  - `originalSourceUrl`: ニュース元記事のURL
  - `noteArticleUrl`: 3本をまとめた note 記事のURL
  - `hpArticleUrl`: HP内の個別記事URL

## 6. 各部品の詳細

### ① 記事データ契約

- **役割**: HP が受け取る「1本の記事」の最小項目を決める
- **最低限必要な項目**:
  - タイトル
  - `originalSourceUrl`（ニュース元記事URL）
  - `noteArticleUrl`（3本まとめの note 記事URL）
  - 公開日時
  - 朝/夜区分
  - 本文（note 記事内の ① / ② / ③ の該当解説文）
  - タグ
  - ソース名
- **確定済み**: `CortexFlow2.0` は JSON（配列形式）と個別 Markdown 両方を出力する。HP は JSON を読む。

#### ①-1. 最小データ項目（おすすめ）

| 項目名 | 必須 | 意味 | 備考 |
|--------|------|------|------|
| `articleId` | 必須 | 1本の記事を一意に識別するID | 例: `2026-03-18_morning_1`（CortexFlow2.0の実際の出力形式） |
| `title` | 必須 | HP上で表示する記事タイトル | note内の①/②/③の見出しに対応 |
| `publishedAt` | 必須 | HP上の公開日時 | ISO形式推奨 |
| `edition` | 必須 | 朝便か夜便か | `morning` / `evening` |
| `orderInEdition` | 必須 | その便の中で何本目か | `1` / `2` / `3` |
| `body` | 必須 | HPで表示する解説本文 | note記事内の該当解説文を1本分として切り出したもの |
| `originalSourceUrl` | 必須 | ニュース元記事のURL | noteのURLではない |
| `sourceName` | 必須 | 元ソース名 | 例: `OpenAI`, `Anthropic` |
| `tags` | 必須 | 検索・整理用のタグ | 空配列は可 |
| `category` | 推奨 | 一覧整理用の分類 | 例: `model`, `product`, `policy` |
| `summary` | 推奨 | 一覧やOGPで使う短い要約 | 本文の短縮版 |
| `noteArticleUrl` | 任意 | その3本を含む note 記事のURL | note公開後に追記でもよい |

#### ①-2. 運用上の考え方

- `originalSourceUrl` は **ニュース元記事** のURL
- `noteArticleUrl` は **3本まとめの note 記事** のURL
- HPでは **`body` を主本文として読む**
- `noteArticleUrl` は補助導線なので、HP公開時点で未確定なら空でもよい

#### ①-3. 推奨データ例

```json
{
  "articleId": "2026-03-18_morning_1",
  "title": "OpenAIの新API発表が示す次の競争軸",
  "publishedAt": "2026-03-18T06:30:00+09:00",
  "edition": "morning",
  "orderInEdition": 1,
  "body": "ここに1本分の解説本文が入る。",
  "originalSourceUrl": "https://example.com/original-news",
  "sourceName": "OpenAI",
  "tags": ["OpenAI", "API", "AIニュース"],
  "category": "model",
  "summary": "新API発表の要点と競争上の意味を短くまとめた要約。",
  "noteArticleUrl": "https://note.com/xxxxx/n/xxxxx"
}
```

### ② 記事取り込み層

- **役割**: `CortexFlow2.0` 由来の個別記事を読み込み、一覧や詳細に使える形へ整える
- **責務**: 読み込み、並び順、日付・タグ整形、欠損チェック
- **前提**: まずは静的サイト前提で扱えること

### ③ 個別記事表示

- **役割**: note では3本のうちの1本を、HPでは独立ページとして読めるようにする
- **最低限必要な見せ方**:
  - タイトル
  - 本文
  - `originalSourceUrl`
  - 公開日時
  - 朝/夜区分
  - タグ
  - 必要に応じて `noteArticleUrl` へのリンク
- **明確にしないこと**: HPから note 記事へ遷移させるだけのリンク集にはしない

### ④ 一覧・アーカイブ導線

- **役割**: 記事を後から探せるようにする
- **最低限必要な見せ方**: 新着一覧、日付順、タグ・カテゴリ・月別アーカイブ
- **後回しにできるもの**: 高機能検索

### ⑤ ブランドページ群

- **役割**: 記事以外に、自分が何者で何を目指しているかを伝える
- **最低限必要なページ**: About / Profile / Vision

### ⑥ 媒体導線

- **役割**: note / X / Slack と HP の役割分担をページ上で迷わず伝える
- **前提方針**: HP = 個別記事の正本、note = 3本まとめ、X / Slack = 配信導線
