# サービス概要（ドメイン前提）

## 0. Status

- Status: APPROVED

## 1. サービス基本情報

- **サービス名**: My Home Page（仮）
- **サービスの位置づけ**: 個人ブランドサイト兼、AI解説記事の検索しやすい蓄積基盤
- **目的**: 毎日作成しているAIニュース解説記事を、朝3本・夜3本の配信単位とは切り分けて、1本ずつ独立して掲載・検索できるようにする。同時に、将来の医療AI起業に向けて、プロフィール、思想、活動、実績を集約する公式サイトとして育てる。
- **対象ユーザー**:
  - 日々のAIニュースを、要点と解説つきで素早く把握したい読者
  - 過去に見た記事をタイトルやタグで探したい既存読者
  - 医療AI領域での発信者・事業家としての自分を知りたい見込み顧客、協業先、採用候補、メディア関係者
- **想定利用シーン**:
  - 読者が「この前見たあの記事どこだっけ」と思ったときに、タイトル・日付・タグからすぐ探す
  - X や note から流入した人が、関連記事やプロフィールをたどって理解を深める
  - 将来、医療AIに関する活動実績や構想を見せる名刺代わりのサイトとして使う

## 2. 価値提案（提供価値）

- 朝3本・夜3本のまとめ配信とは別に、各記事を独立ページとして保存し、後から探しやすくする
- 各記事にタイトル、公開日時、元ニュースURL、要約、解説を持たせ、情報の意味を短時間で理解できるようにする
- note・X・Slack の既存運用を崩さず、HPのみを個別記事の掲載先として追加できるようにする
- ただの記事置き場ではなく、医療AI起業準備中の個人ブランドを伝える公式サイトとして機能させる
- カテゴリー、タグ、アーカイブ、検索により、日々の発信を長期的な知的資産へ変える

## 3. コア機能（MVP）

- 個別記事ページ
  - 1日6本の記事を、朝便・夜便のまとまりとは別に1本ずつ公開する
  - 各記事は `タイトル` `公開日時` `originalSourceUrl` `noteArticleUrl` `要約` `解説` `タグ` `カテゴリー` を持つ
- 記事一覧ページ
  - 最新順に個別記事を一覧表示する
  - 朝便・夜便の区別は表示できるが、保存単位の主役は個別記事とする
- タグ・カテゴリー・日付アーカイブ
  - 過去記事をあとから探しやすくする
- 検索機能
  - 少なくともタイトル検索、できればタグ・本文要約も対象にする
- About / Profile / Vision ページ
  - 自分の経歴、発信テーマ、医療AIで何を目指しているかを示す
- 外部導線
  - note、X、Slack、問い合わせ導線を設ける
- SEO / OGP / RSS
  - 記事単位で共有しやすくし、検索流入を取りにいく

## 4. 情報設計の原則

- **配信単位と保存単位を分ける**
  - 配信単位: 朝3本、夜3本
  - 保存単位: 1本ずつの個別記事
- **HPを資産の母艦にする**
  - HPは個別記事の正本として蓄積する
  - note は朝便・夜便のまとめ記事、X と Slack は既存の配信導線として使う
  - HPは個別記事ページを持ち、note のまとめ記事へ飛ばすだけの構成にはしない
- **ニュースだけのサイトにしない**
  - 将来の医療AI関連コンテンツを追加できる情報設計にする
- **探しやすさを最優先にする**
  - きれいさより、タイトル・日付・タグですぐ見つかることを優先する

## 5. 代表ユースケース

- UC-1: 読者が以前読んだ記事を探したい。トップページまたは検索からタイトルの一部を入力し、候補一覧から該当記事を開き、元ニュースURLと解説を再確認する
- UC-2: 読者がXまたはnoteから個別記事に流入する。記事詳細を読み、関連タグや関連記事をたどり、最終的にプロフィールや他の発信へ進む
- UC-3: 著者が `20260222_CortexFlow2.0` で生成した個別記事データをもとに、HPには1本ずつ掲載し、noteでは朝便・夜便のまとめ記事として配信する。X と Slack は既存の運用方式を維持する

## 6. 成功指標（KPI）

- 個別記事掲載数: 1日6本のうち、安定して何本をHPへ個別掲載できたか
- 記事到達性: 公開から30日以上経った記事にも一定の閲覧が発生しているか
- 検索利用率: タイトル検索、タグ遷移、アーカイブ経由の閲覧割合
- ブランド到達: プロフィール、Vision、医療AI関連ページの閲覧数
- 指名流入: 自分の名前、ブランド名、活動名での流入
- 品質指標: Lighthouse の Performance / SEO / Accessibility で 90 点以上

## 7. 制約・非機能要件

- **法務/コンプライアンス**: 外部ニュースは出典を明示し、引用要件を満たす。医療AI関連の記述は、診断・治療の助言と誤認されない表現に注意する
- **媒体方針/SEO方針**: HPは個別記事の正本、noteは朝便・夜便のまとめ記事とする。完全同一ページの量産は避け、必要に応じて canonical を含めた運用方針を整理する
- **セキュリティ**: まずは静的サイト構成を基本とし、サーバーサイドの攻撃面を増やさない。問い合わせは外部フォーム利用を優先する
- **データ/プライバシー**: 個人情報を自前で保持しない。アクセス解析導入時はプライバシーポリシーを用意する
- **パフォーマンス**: モバイル回線でも主要記事が速く開くこと。LCP 2.5 秒未満を目標にする
- **アクセシビリティ**: 文字サイズ、見出し構造、コントラスト比を満たし、長文でも読みやすくする
- **拡張性**: 将来、医療AI、プロジェクト、講演、実績ページを無理なく追加できる構造にする

## 8. 技術スタック（初期想定）

- **Core**: Next.js 14 以降（App Router）+ TypeScript
- **UI**: Tailwind CSS。動きは控えめにし、読みやすさを優先する
- **コンテンツ管理**:
  - 初期は Markdown / MDX ベース
  - 記事データは個別記事として構造化して保持する
  - `20260222_CortexFlow2.0` 側で個別記事を正規単位として出力し、HPはそれを受け取る構造に寄せる
- **検索**:
  - 初期は静的検索で足りる方式を選ぶ
  - 記事増加後に Pagefind や外部検索への拡張を検討する
- **配信/公開**: GitHub + Vercel などの静的配信基盤
- **テスト**: 最低限、一覧表示、記事詳細、検索導線のスモークテストを持つ

## 8.1. リスクと未確定事項

- **ブランド名未確定**: サイト名を個人名ベースにするか、メディア名を立てるか未決定
- **CortexFlow連携方式未確定**: `20260222_CortexFlow2.0` 側で個別記事データをどの形式で保持・出力するかは未決定
- **項目定義の最終化**: `originalSourceUrl` と `noteArticleUrl` をどう保持し、HP個別記事URLとどう結びつけるかは実装前に固める
- **検索方式未確定**: 初期は軽量実装で良いが、記事数増加時の拡張方針は要検討
- **コンテンツ範囲の線引き**: AIニュース解説、医療AIの構想、個人的な論考の境界をどう見せるか未決定
- **継続運用負荷**: 毎日6本の個別掲載を追加しても、今の運用負荷に無理がないか検証が必要
